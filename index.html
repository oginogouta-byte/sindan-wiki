Overview

全体設計

スキルプラスのAI活用度診断ツール。100問の回答から個別最適化されたレポートを生成し、UTAGEまたはX（引用RT）経由でURLを届ける。レポート末尾でLINE公式アカウントへ誘導してリストを獲得する。LINEでのレポート送付は行わない。

System Flow

システムフロー

診断完了 → 結果を見るためのCVアクション → レポート送付の流れ。結果はCVトリガー後に初めて届く設計。

① 診断HTML（100問回答）
ユーザーが100問に回答。完了時に診断ID + 回答データをGoogle Apps ScriptへPOST。この時点でスコア・結果は表示しない。
② GAS：スコア計算 → Sheets保存
POST受信後、4軸スコア計算 → サブカテゴリ内訳算出 → 分散計算 → GAP計算 → 整合性チェック → 5段階レベル算出（3層変数生成）→ Sheetsに1行書き込み。診断IDをキーとして管理。結果はユーザーにはまだ届かない。
③ UTAGEサンクスページ（部分表示 → CVフック）
GASがスコア計算完了後、主要数値をURLパラメータに付与してリダイレクト。ページ上部にAI戦闘力・上位X%・4軸スコア（数値）を表示。詳細評価テキスト部分はサンプル文＋CSSブラーで「中身はあるが見えない」状態にし、CVへの動機を最大化する。
実装：UTAGEカスタムJS（setTimeout + URLSearchParams）でパラメータ読み取り → DOM書込。外部API呼び出しなし・同時アクセス安全（検証済み）。
④ CVトリガー分岐（2案）
案A（500円決済）：UTAGEの決済フォーム → 決済完了 → メールアドレス・氏名取得 → GASがSheets照合 → Mailchimpでレポートの個別URLをメール送付。
案B（X引用RT）：固定ツイートのリンク → 引用RTを投稿 → X APIでユーザー特定・診断IDと照合 → XのDMでレポートURL送付。
⑤ GAS：診断ID × ユーザーID 紐づけ
いずれのルートでも診断IDとユーザー情報をGASで照合・Sheetsに記録。
⑥ Claude API：レポート生成
SheetsからLayer1〜3の3層変数を取得し、Claude APIに単一リクエストとして渡してJSON一括生成（1回のみ）。生成結果をSheetsに保存 → 個別URL発行。
⑦ レポート送付 → LINE登録CTA（リスト獲得）
案A：Mailchimpでメール送付 / 案B：XのDMで送付。レポート末尾にLINE公式アカウントへの登録CTAを設置し、リストを獲得する。LINEでの送付は行わない。
ポイント：AI生成はレポート閲覧時ではなくCVトリガー後の1回のみ実行。コスト固定・表示速度を確保。生回答データ（100問）はClaudeに渡さず、3層変数に集約してから渡す。
Scoring

スコアリング設計

各観点25問 × 最大5点 = 各観点125点。4観点の内部合計（最大500点）を非線形変換し、AI戦闘力として表示する。

戦闘力の算出式（軸別加算型）：
各軸トークン = round( (軸スコア ÷ 125)³ × 249,999 ) AI戦闘力（総合）= 知識力 + 指示力 + 活用力 + 思考力（各軸トークンの合計）

4軸それぞれで3乗変換し合算。各軸最大 249,999トークン、総合最大 999,999トークン。GASがトークン変換まで計算しREPORT_DATAに注入。フロントは表示のみ。

縦スクロール型・個別レポート。性格分類ではなく「市場内の能力位置づけ」をパフォーマンス分析として提示する。

設計思想：タイプ名・ランクなし。「市場の中で自分がどこにいるか」を上位X%で即時把握させ、サブカテゴリの数値と言語化の組み合わせで「このレポートは自分のことを全部分かっている」という体験を作る。デザイン品質がコンテンツ品質に見える。診断ページと同水準のビジュアルで、業界の基準を作りにいく。
コンバージョン設計：レポート体験の質そのものがLINE登録のドライバー。低スコア層は「情報が欲しい」、高スコア層は「成長意欲が刺激される」という2つのパスで自然に誘導する。CTAは「1on1誘導」ではなく「個別最適化された情報をLINEでお届け」。

UX Design

UX設計

縦スクロール型・パフォーマンス分析フレームで構成する。

設計方針：情報密度を担保するため縦スクロール型を採用。セクション単位でコンテンツを区切り、ビジュアルゾーン（FV〜2）→ テキスト分析ゾーン（3〜6）→ CTA の順で自然に読み進められる構成にする。
デザイン品質方針：診断ページ（index.html）のビジュアル水準を前提とする。一般ユーザーはAI診断の良し悪しを判断する軸を持っていないため、デザインの先進性・精緻さが「このサービスはすごい」という認知を直接作る。市場にほぼ競合が存在しない今、このレポートが業界の基準になる。レーダーチャート・スコアバーはデフォルトライブラリを使わず、診断ページと同世界観のカスタムビジュアルで構築する。
縦スクロール型
スワイプで下にスクロールするシームレスな1ページ構成。情報密度を高く保てる。
セクション区切り
H1見出しごとにセクションを区切り「今何の話をしているか」を明示する。ビジュアル要素と文章のコンビネーション。
条件付き表示
GAPセクションは有意なGAPがある場合のみ表示。不整合0件の場合も1文で示す。コンテンツ量がユーザーによって変わる設計。
モバイルファースト
基本設計は375px〜のスマートフォン縦向きを想定。タップ・スクロール操作を基準とする。

Claude API

Claude API生成設計

データ受信時（1回のみ）に実行。生成済みテキストをSheetsに保存し、レポートHTMLに差し込む。

入力データ（確定）：3層変数アーキテクチャをそのまま渡す。Layer1（軸スコア・5段階レベル・合計スコア・ランク・上位X%）+ Layer2（サブカテゴリ12本・分散4本）+ Layer3（不整合グループ名・件数・min/max）+ Cross-axis GAP（ki / at / kt の3本）。100問の生回答データは渡さない（ノイズが多く生成品質が下がるため）。
セクション 出力フィールド 形式 Sheets保存列
診断サマリー summary 文章 100〜120字 DK列
軸別評価 × 4 eval 文章 〜80字 DL〜DO列
strengths 配列 2点固定・各〜20字
weaknesses 配列 2点固定・各〜20字
growth 文章 〜80字
あなたの特徴 gaps[] 配列（条件付き）description 〜80字 DP列

これからのあなたへ priority_action 短文 〜30字 DQ列
explanation 文章 150〜180字
形式選択の根拠：
  • 文章：意味・論理・感情の流れが必要な要素（eval / growth / gaps / explanation）
  • 配列（各〜20字）：strengths / weaknesses はスキャン・タグ的消化が目的。4軸繰り返しなので短く統一して認知負荷を抑える
  • 短文（〜30字）：priority_action は「1点に絞った行動指針」としてデザイン上強調できる長さに固定

GAPセクション・カード構造（確定）：
  • 表示条件：絶対値 ≥ 25点（正負両方）のペアのみ。最大3本（ki / at / kt）。0本 → セクションごと非表示
  • 概念説明（セクション冒頭1回のみ・確定テキスト）：「4つの能力のうち、25%以上の差がある組み合わせを検出しました。多くの人が気づいていない、あなたのAI活用の弱みがわかります。」― ペアごとには置かない
  • 大なり小なり見出し：GASがスコア比較で動的生成。例：「知識力 > 指示力」
  • スコアビジュアル：軸スコア ÷ 125 × 100 = %バー。2軸並列 ＋ 差分%を数値で明示

Evaluation Structure

評価の3層構造

個別最適化レポートを生み出すための3つの評価レイヤー。MBTIのような事前テキストマッピングではなく、Claude APIが3層の文脈から都度生成する。

根本方程式：アウトプット = 思考フレームの質 × AI実行能力
AIは「増幅装置」であって「補完装置」ではない。弱い思考を強くはしないが、強い思考を大量に実行してくれる。
AIが提供する3つの価値
① 速度と量— 自分の思考を分身として再現・量産する（100人でやっていたことを1人で）
② スキル実行— 自分にないスキルセットを実行できる（デザイン・コード・法務…）
③ 思考の外部化— 自分の思考フレームをAIに設計・再現させる
Layer 1 — 数値
スコア（軸スコア + サブカテゴリ内訳）
各観点スコア（/125）をサブカテゴリに分解することで「知識力が高い」ではなく「ツール知識は高いが本質理解が低い」という粒度を得る。サブスコアの組み合わせパターンがClaude APIへの入力文脈になる。
例：知識力 88/125 → ツール知識（高）/ 本質・限界理解（低）/ 動向・発展把握（中）
Layer 2 — パターン
サブカテゴリ内訳 + 軸内分散（どうやって到達したか）
軸スコア合計だけでは見えない「偏り」を検出する。「指示設計（高）× 基本指示（低）」と「全サブ均等中位」は同じ合計でもClaudeへの入力文脈が変わる。分類タイプ名はGAS内部計算のみ。Claudeにはサブスコア12本と分散4本を直接渡す。
例：指示力 72/125 → 基本指示（低）/ 反復改善（中）/ 指示設計（高）→ 「型は持っているが入門層で詰まっている」と解釈
Layer 3 — 矛盾
回答の一貫性チェック（同テーマの回答のブレを検出）
指示力・活用力・思考力の3軸×3グループ=9グループで「同じ行動テーマの設問に対して回答が矛盾している」状態を検出。バッジ3×3で視覚化し、Claude APIが80〜120字の単一評価文を生成。
例：思考力のバッジが1/3 → 「AIの出力確認や判断管理に関して、場面によって行動がバラついている可能性があります」

2軸圧縮では消えてしまう。4軸個別評価でのみ検知できるパターン。

A高 × T低 ⚠️
活用力≥87 かつ 思考力≤49
過信実行型 — 最も危険なパターン AIを大量に使っているが出力の検証・品質管理が追いついていない。間違った情報や低品質な成果物をそのまま使い続けるリスクが高い。レポートで最優先に警告すべき。
K高 × I低
知識力≥87 かつ 指示力≤49
知識偏重型 — インプット過多パターン AIの知識・情報収集は豊富だが、具体的な指示設計・活用に変換できていない。インプットを行動に変える力が課題。
K高 × T低
知識力≥87 かつ 思考力≤49
知識過信型 — 盲点パターン AIについてよく知っているが、出力の検証習慣が薄い。「知っているから大丈夫」という思い込みが精度リスクを生む。
K高・T高 × A低
K≥80 かつ T≥80 かつ A≤49
慎重未実践型 — 理解はある、手が動かないパターン 知識もあり出力検証もできるが、実際の活用量が圧倒的に少ない。理解と実践の間の心理的な壁が存在する。

API Input

Claude API 入力変数設計

タイプ名ではなく生の数値をClaudeに渡す。3層の変数群がどんなパターンにも対応する個別最適化コンテキストを生成する。

設計原則：変数が揃っていれば、Claudeはどんな数値の組み合わせにも意味を見出せる。タイプ判定の精度より変数設計の精度が出力品質を決める。

Layer 1 ── 位置の把握（どこにいるか）

変数名内容算出元
{{rank}}ランク（S / A / B / C / D）Sheets DD列
{{total_score}}合計スコア（/500）Sheets DB列
{{percentile}}全体の上位X%（例：12.4）Sheets DG列（PERCENTRANK）
{{k_score}} / {{i_score}} / {{a_score}} / {{t_score}}各軸スコア（/125）Sheets CX〜DA列
{{k_level}} / {{i_level}} / {{a_level}} / {{t_level}}各軸の5段階レベル（1〜5）GAS計算（スコアをレンジで変換）
5段階換算：L1=0〜24点 / L2=25〜49点 / L3=50〜74点 / L4=75〜99点 / L5=100〜125点

Layer 2 ── 内訳の分析（どうやって到達したか）

同じ軸スコアでもサブカテゴリの内訳が違えばレポートは別物になる。中間層の個別最適化もこの層が担う。

変数名内容算出元
{{k_sub1}} / {{k_sub2}} / {{k_sub3}} 知識力サブ（①ツール把握 /11問 ②仕組み理解 /8問 ③先端応用 /6問） GAS：対象設問を合算
{{i_sub1}} / {{i_sub2}} / {{i_sub3}} 指示力サブ（①基本指示 /11問 ②対話改善 /8問 ③指示設計 /6問） GAS：対象設問を合算
{{a_sub1}} / {{a_sub2}} / {{a_sub3}} 活用力サブ（①日常活用 /11問 ②スキル補完 /8問 ③組織展開 /6問） GAS：対象設問を合算
{{t_sub1}} / {{t_sub2}} / {{t_sub3}} 思考力サブ（①出力確認 /11問 ②精密検証 /8問 ③リスク判断 /6問） GAS：対象設問を合算

分散の読み方：同じ知識力65点でも「Sub1=高・Sub3=低（偏り大）」と「全Sub均等（偏り小）」では評価が異なる。偏りが大きい人には「どこが強くてどこが弱いか」、偏りが小さい人には「全体的な底上げ」という異なるフィードバックが生成される。

Gap Analysis

GAP分析設計

4軸間の乖離を検出し「認知チェーンの詰まり」を可視化するロジック。3ペア × 正逆方向で最大6パターンの乖離を検知する。

GAPペアと設計意図

ペア計算式正方向（+25以上）の意味逆方向（−25以下）の意味状態
K-I GAP 知識力 − 指示力 知識はあるのに指示で活かせない（知識の空転） 指示力が突出・知識の裏付けが薄い 逆方向未定義
A-T GAP 活用力 − 思考力 使いまくっているが思考・検証が追いついていない（過信実行型） 思考・検証過剰・実行が伴わない（評論型） 逆方向未定義
K-T GAP 知識力 − 思考力 知識があるのに出力検証・判断が弱い（思考処理不全） 思考力は高いが知識基盤が薄い 逆方向未定義
検知しない3ペアについて：I-A / I-T / A-K は現在対象外。選定した3ペアは「知識→指示→活用→検証」という認知チェーンの詰まり場所を特定する意図で選ばれている。残りのペアを将来追加するかは後工程で判断する。

⚠️ 単位の不一致：分散 vs GAP

要対応：Claude API へ渡す変数に単位が混在している。プロンプトか GAS のどちらかで統一する必要がある。
変数種別計算方法スケール保存列
各軸の分散 Sub最大率 − Sub最小率
各Subスコアをそれぞれの最大値で割った比率の差		 0〜1（比率差） ED〜EG

案A：GAPも比率化 (K-I) ÷ 125 で 0〜1 に正規化 分散と同スケール・プロンプトがシンプル 点数差の直感的な感覚が消える
案B：プロンプトで単位を明記 「分散は比率差(0〜1)、GAPは点数差(/125)」と記述 既存の計算ロジックを変更しなくてよい プロンプトが長くなる・誤解リスクが残る
案C：分散も点数化 Sub率差 × 軸最大スコア(125)で点数換算 GAP と同スケールで直感的 Sub の問数が軸によって異なる（11/8/6問）ため Max が揃わない

注意（検証済み）：UTAGEはVue.jsでクライアントサイドレンダリングしている。スクリプトがページロード時に即実行するとDOMがまだ存在せず、getElementById が null を返す。setTimeout(fn, 300) で300ms待ってから実行すること。

UTAGEカスタムJS（貼り付け用）

setTimeout(function () {
  var p = new URLSearchParams(window.location.search);

  function set(id, val) {
    var el = document.getElementById(id);
    if (el) el.textContent = val;
  }
  function setWidth(id, pct) {
    var el = document.getElementById(id);
    if (el) el.style.width = pct + '%';
  }

  var total = parseInt(p.get('total') || 0);
  var pct   = parseFloat(p.get('pct')   || 0);
  var k = parseInt(p.get('k') || 0);
  var i = parseInt(p.get('i') || 0);
  var a = parseInt(p.get('a') || 0);
  var t = parseInt(p.get('t') || 0);

  set('st-total-tokens', total.toLocaleString());
  set('st-percentile',   pct >= 10 ? pct.toFixed(1) + '%' : pct.toFixed(2) + '%');

  set('st-score-k', k);  setWidth('st-gauge-k', Math.round(k / 125 * 100));
  set('st-score-i', i);  setWidth('st-gauge-i', Math.round(i / 125 * 100));
  set('st-score-a', a);  setWidth('st-gauge-a', Math.round(a / 125 * 100));
  set('st-score-t', t);  setWidth('st-gauge-t', Math.round(t / 125 * 100));

}, 300);
