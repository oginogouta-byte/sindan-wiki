System Flow

システムフロー

診断完了 → 結果を見るためのCVアクション → レポート送付の流れ。結果はCVトリガー後に初めて届く設計。

① 診断HTML（100問回答）
ユーザーが100問に回答。完了時に診断ID + 回答データをGoogle Apps ScriptへPOST。この時点でスコア・結果は表示しない。
② GAS：スコア計算 → Sheets保存
POST受信後、4軸スコア計算 → サブカテゴリ内訳算出 → 分散計算 → GAP計算 → 整合性チェック → 5段階レベル算出（3層変数生成）→ Sheetsに1行書き込み。診断IDをキーとして管理。結果はユーザーにはまだ届かない。
③ UTAGEサンクスページ（部分表示 → CVフック）
GASがスコア計算完了後、主要数値をURLパラメータに付与してリダイレクト。ページ上部にAI戦闘力・上位X%・4軸スコア（数値）を表示。詳細評価テキスト部分はサンプル文＋CSSブラーで「中身はあるが見えない」状態にし、CVへの動機を最大化する。
実装：UTAGEカスタムJS（setTimeout + URLSearchParams）でパラメータ読み取り → DOM書込。外部API呼び出しなし・同時アクセス安全（検証済み）。
④ CVトリガー分岐（2案）
案A（500円決済）：UTAGEの決済フォーム → 決済完了 → メールアドレス・氏名取得 → GASがSheets照合 → Mailchimpでレポートの個別URLをメール送付。
案B（X引用RT）：固定ツイートのリンク → 引用RTを投稿 → X APIでユーザー特定・診断IDと照合 → XのDMでレポートURL送付。
⑤ GAS：診断ID × ユーザーID 紐づけ
いずれのルートでも診断IDとユーザー情報をGASで照合・Sheetsに記録。
⑥ Claude API：レポート生成
SheetsからLayer1〜3の3層変数を取得し、Claude APIに単一リクエストとして渡してJSON一括生成（1回のみ）。生成結果をSheetsに保存 → 個別URL発行。
⑦ レポート送付 → LINE登録CTA（リスト獲得）
案A：Mailchimpでメール送付 / 案B：XのDMで送付。レポート末尾にLINE公式アカウントへの登録CTAを設置し、リストを獲得する。LINEでの送付は行わない。
ポイント：AI生成はレポート閲覧時ではなくCVトリガー後の1回のみ実行。コスト固定・表示速度を確保。生回答データ（100問）はClaudeに渡さず、3層変数に集約してから渡す。
Scoring

スコアリング設計

各観点25問 × 最大5点 = 各観点125点。4観点の内部合計（最大500点）を非線形変換し、AI戦闘力として表示する。

戦闘力の算出式（軸別加算型）：
各軸トークン = round( (軸スコア ÷ 125)³ × 249,999 ) AI戦闘力（総合）= 知識力 + 指示力 + 活用力 + 思考力（各軸トークンの合計）

4軸それぞれで3乗変換し合算。各軸最大 249,999トークン、総合最大 999,999トークン。GASがトークン変換まで計算しREPORT_DATAに注入。フロントは表示のみ。

Report Design

レポート全体構成

縦スクロール型・個別レポート。性格分類ではなく「市場内の能力位置づけ」をパフォーマンス分析として提示する。

設計思想：タイプ名・ランクなし。「市場の中で自分がどこにいるか」を上位X%で即時把握させ、サブカテゴリの数値と言語化の組み合わせで「このレポートは自分のことを全部分かっている」という体験を作る。デザイン品質がコンテンツ品質に見える。診断ページと同水準のビジュアルで、業界の基準を作りにいく。
コンバージョン設計：レポート体験の質そのものがLINE登録のドライバー。低スコア層は「情報が欲しい」、高スコア層は「成長意欲が刺激される」という2つのパスで自然に誘導する。CTAは「1on1誘導」ではなく「個別最適化された情報をLINEでお届け」。

Evaluation Structure

評価の3層構造

個別最適化レポートを生み出すための3つの評価レイヤー。MBTIのような事前テキストマッピングではなく、Claude APIが3層の文脈から都度生成する。

根本方程式：アウトプット = 思考フレームの質 × AI実行能力
AIは「増幅装置」であって「補完装置」ではない。弱い思考を強くはしないが、強い思考を大量に実行してくれる。
AIが提供する3つの価値
① 速度と量— 自分の思考を分身として再現・量産する（100人でやっていたことを1人で）
② スキル実行— 自分にないスキルセットを実行できる（デザイン・コード・法務…）
③ 思考の外部化— 自分の思考フレームをAIに設計・再現させる
Layer 1 — 数値
スコア（軸スコア + サブカテゴリ内訳）
各観点スコア（/125）をサブカテゴリに分解することで「知識力が高い」ではなく「ツール知識は高いが本質理解が低い」という粒度を得る。サブスコアの組み合わせパターンがClaude APIへの入力文脈になる。
例：知識力 88/125 → ツール知識（高）/ 本質・限界理解（低）/ 動向・発展把握（中）
Layer 2 — パターン
サブカテゴリ内訳 + 軸内分散（どうやって到達したか）
軸スコア合計だけでは見えない「偏り」を検出する。「指示設計（高）× 基本指示（低）」と「全サブ均等中位」は同じ合計でもClaudeへの入力文脈が変わる。分類タイプ名はGAS内部計算のみ。Claudeにはサブスコア12本と分散4本を直接渡す。
例：指示力 72/125 → 基本指示（低）/ 反復改善（中）/ 指示設計（高）→ 「型は持っているが入門層で詰まっている」と解釈
Layer 3 — 矛盾
回答の一貫性チェック（同テーマの回答のブレを検出）
指示力・活用力・思考力の3軸×3グループ=9グループで「同じ行動テーマの設問に対して回答が矛盾している」状態を検出。バッジ3×3で視覚化し、Claude APIが108〜120字の単一評価文を生成。
例：思考力のバッジが1/3 → 「AIの出力確認や判断管理に関して、場面によって行動がバラついている可能性があります」

API Input

Claude API 入力変数設計

タイプ名ではなく生の数値をClaudeに渡す。3層の変数群がどんなパターンにも対応する個別最適化コンテキストを生成する。

設計原則：変数が揃っていれば、Claudeはどんな数値の組み合わせにも意味を見出せる。タイプ判定の精度より変数設計の精度が出力品質を決める。

Layer 1 ── 位置の把握（どこにいるか）

変数名内容算出元
{{rank}}ランク（S / A / B / C / D）Sheets DD列
{{total_score}}合計スコア（/500）Sheets DB列

Overview

概要・設計方針（確定）

GASフェーズ1がURLパラメータ付きで UTAGEサンクスページへリダイレクト。UTAGE上で report.html の FV〜AI戦闘力セクション相当のHTMLを表示し、カスタムJSがURLパラメータの値を注入する。ユーザーがCV行動（Xリポスト / 決済）を行っている「余白の時間」の間に GASフェーズ2 が Claude API でレポートを生成し、CV完了後にフル report.html の URL を届ける。

設計の核心：CV行動が「生成の余白」になる
ユーザーがサンクスページでXリポストや決済手続きをしている時間 = GASが Claude API を呼んでレポートを生成する時間。ユーザーが待たずに済み、CV完了時点でレポートが仕上がっている。この「余白の時間」を活かすために UTAGE を経由させる。

setTimeout(function () {
  var p   = new URLSearchParams(window.location.search);
  var MAX = 249999;

  // 生スコア(0〜125) → トークン値(0〜249,999) に変換
  function toToken(raw) {
    return Math.round(Math.pow(parseInt(raw || 0) / 125, 3) * MAX);
  }
  function fmt(n) {
    // sc-comma スパンを挟んだカンマ区切り（report.html の表示ルールに合わせる）
    return n.toLocaleString('ja-JP').replace(/,/g, ',');
  }
  function setHTML(id, v) { var el = document.getElementById(id); if (el) el.innerHTML = v; }
  function setText(id, v) { var el = document.getElementById(id); if (el) el.textContent = v; }
  function setWidth(id, w) { var el = document.getElementById(id); if (el) el.style.width = w + '%'; }

  var total = parseInt(p.get('total') || 0);
  var pct   = parseFloat(p.get('pct') || 0);
  var kT = toToken(p.get('k'));
  var iT = toToken(p.get('i'));
  var aT = toToken(p.get('a'));
  var tT = toToken(p.get('t'));

  // --- FV 注入 ---
  // ※ FV の canvas（ピラミッド・アークゲージ）は要別途対応（下記 callout 参照）
  setText('fv-rank-val', pct >= 10 ? pct.toFixed(1) + '%' : pct.toFixed(2) + '%');

  // --- Section 1（AI戦闘力）注入 ---
  setHTML('score-knowledge',   fmt(kT));
  setHTML('score-instruction', fmt(iT));
  setHTML('score-application', fmt(aT));
  setHTML('score-thinking',    fmt(tT));
  setText('pct-knowledge',   Math.round(kT / MAX * 100) + '%');
  setText('pct-instruction', Math.round(iT / MAX * 100) + '%');
  setText('pct-application', Math.round(aT / MAX * 100) + '%');
  setText('pct-thinking',    Math.round(tT / MAX * 100) + '%');
  setWidth('gauge-knowledge',   kT / MAX * 100);
  setWidth('gauge-instruction', iT / MAX * 100);
  setWidth('gauge-application', aT / MAX * 100);
  setWidth('gauge-thinking',    tT / MAX * 100);

  // --- .score-summary をブラー ---
  var summary = document.querySelector('.score-summary');
  if (summary) { summary.style.filter = 'blur(6px)'; summary.style.pointerEvents = 'none'; }

}, 300);

設計方針

プロンプトは「禁止リスト」ではなく「前提と目的の正確な伝達」で制御する。Claudeの自然な能力を活かす設計。

① 計算された診断結果であることを前提として渡す
この診断は100問の回答データから3層変数を算出した個別最適化の結果。 あいまいな推測ではなく、数値が証明している事実を提示する場である。 この文脈をClaudeに正確に渡せば、断言的な文章は自然に出てくる。
② 数値を根拠に、そこから読める情報を提示する
ビジュアルが数値を見せる。テキストはその数値が何を意味するかを提示する。 「あなたはこうです。この数値がそれを証明しています。」という構造。 推測・共感・励ましではなく、客観的な事実の提示が読者に刺さる。
③ 優れている点・劣っている点を両方言い切る
ユーザーは市場のAI診断に懐疑的。フラットな文は「誰でも同じ」と読まれる。 強みは言い切り、弱みも言い切る。その精度がこの診断の価値になる。
プロンプト設計の原則：禁止リストではなく前提と目的で制御
「〜と書くな」という制約を並べるより、Claudeが正しい文脈を理解していれば自然に正しい文章になる。 データ・目的・出力スキーマを正確に渡すことが最も重要。
